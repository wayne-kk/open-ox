# 功能 V1.0.0 使用说明书

# 跨端前端界面智能生成与美化渲染系统 v1.0.0

---

# **1. 简介**

## **1.1 软件名称和用途**

### **1.1.1 软件名称**

跨端前端界面智能生成与美化渲染系统 v1.0.0

### **1.1.2 用途**

本系统面向 **跨端 Web 前端界面**（基于 Next.js / React，可扩展至移动 Web 与同源 H5）的智能化生产，为用户提供从自然语言需求到 **可运行、可预览、可迭代美化** 的完整界面交付环境。主要用途包括：

（1）**界面结构与页面蓝图自动生成**：将用户描述（或参考网站/截图）转化为 `ProjectBlueprint`，包含多页路由、导航模型、各页 Section 区块顺序与交互要点。

（2）**设计系统与美化渲染**：自动匹配或生成 `design-system.md`，并通过 `apply_project_design_tokens` 将主题色、字体、圆角、动效变量写入 `globals.css`（Tailwind CSS v4 CSS Variables），保证全站视觉一致。

（3）**跨页面壳层与分区块实现**：`architect_agent` 生成全局 `layout.tsx` 与 `components/chrome/**`（导航、页脚等）；`page_implement_agent` 并行生成各页 Hero、内容区、表单等 Section 组件，并可按技能库注入 WebGL / GSAP / Framer Motion 等 **美化渲染** 效果。

（4）**实时预览与界面迭代**：Studio 通过 SSE 展示生成步骤；支持 Storage / E2B / 本地多种 **预览渲染**；生成后用自然语言修改导航、Hero、配色等，系统自动搜索组件文件、精确编辑并 `next build` 验证。

---

## **1.2 软件特点和目标用户**

### **1.2.1 软件特点**

（1）**设计系统单真源**：全局样式变量先于页面组件落盘，杜绝各 Section 自行发明颜色与字体导致的风格漂移。

（2）**先壳层、后页面**：全局 Chrome 与 layout 契约先行，分页面 Agent 只填充 `{children}` 区域，避免重复导航栏/页脚。

（3）**美化技能库**：内置 design-system 风格包与 hero/section 特效技能（含 WebGL Hero、滚动叙事等），按设计意图关键词自动路由。

（4）**并行生成与构建自愈**：多页面 Section Agent 并行；`typecheck` + `next build` + 最多 5 轮 `repair_build` 保障界面代码可编译运行。

（5）**参考稿保真**：支持参考截图与 `reference_site_digest`，在 guardrail 约束下复刻版式密度与信息层级（非简单换色）。

### **1.2.2 目标用户**

（1）**市场与运营团队**：快速产出活动页、落地页、官网，无需手写 React 代码。

（2）**独立开发者与小团队**：以自然语言迭代 UI，缩短从原型到可部署站点的时间。

（3）**设计开发培训**：演示 AI 驱动的前端工程化流程与设计系统落地方法。

# **2. 安装和配置**

## **2.1 系统要求**

### **2.1.1 服务端**

| 配置项目 | 要求 |
|----------|------|
| 操作系统 | Linux（Ubuntu 20+）/ macOS |
| CPU / 内存 | 4 核及以上 / 16GB 及以上（推荐 32GB） |
| 运行环境 | Node.js 18+、pnpm 或 npm |
| 可选云服务 | Supabase（项目元数据与预览 Storage）、E2B（云端预览沙箱） |
| 模型 API | OpenAI-compatible 接口（Gemini / GPT 等） |

### **2.1.2 客户端**

| 配置项目 | 要求 |
|----------|------|
| 浏览器 | Chrome 90+、Firefox 88+、Safari 14+、Edge 90+ |
| 网络 | 支持 SSE 流式接收生成进度；预览需访问宿主或 CDN 地址 |

---

## **2.2 安装说明**

（1）克隆源代码至部署目录，安装依赖：

```Bash
pnpm install
# 或 npm install
```

（2）配置环境变量（`.env.local`）：

```Bash
OPENAI_API_KEY=your_key
OPENAI_API_URL=https://your-compatible-endpoint/v1
# 可选：Supabase、Langfuse、E2B、预览后端等
```

（3）启动开发服务：

```Bash
pnpm dev
# 默认 http://localhost:3000
```

（4）浏览器打开首页，在输入框用自然语言描述所需 **界面**（例如：「深色科技风 SaaS 落地页，含 Hero、功能对比、定价表、页脚订阅」）。

（5）可选：上传 **参考 UI 截图** 或选择 `public/skills/` 下的风格指南，以约束美化方向。

（6）提交后进入 Studio（`/studio/{projectId}`），观察 SSE 步骤：需求分析 → 设计系统 → 全局壳层 → 各页 Section → 构建 → 预览。

（7）在预览窗验证交互与视觉效果；如需微调，在对话区用自然语言描述修改（如「Hero 标题加大、主色改为橙色」），系统定位 TSX 文件并自动构建验证。

---

## **2.3 配置说明**

- **模型路由**：`lib/config/models.ts` 为各步骤配置 model id（分析、规划、实现、修复等可分离）。

- **并行度**：`GENERATION_PAGE_IMPLEMENT_CONCURRENCY` 限制同时运行的页面实现 Agent 数量。

- **预览后端**：`OPEN_OX_PREVIEW_BACKEND=storage|e2b|local` 切换预览渲染方式。

- **页面数量护栏**：`GENERATION_MAX_SITE_PAGES`（默认 24）限制极端多页蓝图。

# **3. 系统功能介绍**

## **3.1 界面与控制流程**

系统提供 Web 控制台（首页 + Studio），覆盖「需求输入 → 生成监控 → 预览渲染 → 对话改 UI」全流程。

### **3.1.1 需求输入与项目创建**

- 首页 **HeroPrompt**：自然语言 + 可选风格 skill + 可选截图。
- `POST /api/projects` 立即创建项目记录并跳转 Studio。
- 可选 **Intent Agent** 多轮澄清（页面结构、语气、参考链接），确认后 `commit_generate` 进入流水线。

### **3.1.2 生成进度监控**

- Studio 左侧/中部展示 `buildSteps`（SSE 实时 + DB 持久化，支持刷新恢复）。
- 典型步骤：`analyze_project_requirement` → `plan_project` → `apply_project_design_tokens` → `architect_agent` → `page_implement_agent` → `run_build`。

### **3.1.3 预览渲染**

- 生成成功后嵌入或跳转预览 URL。
- **Storage 模式**：静态 HTML/JS/CSS，适合生产演示。
- **E2B 模式**：隔离沙箱内 `serve` 静态导出产物。

### **3.1.4 界面迭代修改**

- 对话输入视觉/文案修改意图。
- 展示 touched 文件与 **unified diff**（绿增红删）。
- 修改后自动触发预览重建。

---

## **3.2 标准操作步骤（六步）**

### **3.2.1 描述界面需求**

在首页输入产品类型、受众、页面列表、风格关键词（如「极简、医疗蓝、大图 Hero」）。可附参考站 URL 或截图。

### **3.2.2 确认蓝图（可选 Intent 对话）**

系统产出 `ProjectBlueprint`：含 `site.pages[]`、每页 `sections[]`、导航与语言（zh/en）。用户可在 Intent 阶段 yield 确认或补充。

### **3.2.3 自动生成界面**

流水线并行/串行执行：设计意图推断 → 页面规划 → 设计系统 → **globals.css Token 注入** → Architect 壳层 → 各页 Section（含 Hero 技能美化）。

### **3.2.4 构建与渲染校验**

- Scoped TypeScript 诊断；
- `next build`；失败则 `repair_build` 自动改相关 TSX/CSS（≤5 轮）。

### **3.2.5 预览与体验验证**

在预览中检查：响应式布局、导航跳转、动效、对比度与可读性。

### **3.2.6 对话式美化迭代**

用自然语言继续调整；系统遵循 Stop Hook：**先搜索组件 → 再 edit_file → 最后 run_build 通过**。

---

## **3.3 美化渲染机制**

### **3.3.1 Design Tokens 层**

`apply_project_design_tokens` 根据 `design-system.md` 生成/更新 `app/globals.css`，定义 `--color-*`、`--font-*`、`@keyframes` 等，供 Tailwind 语义类引用。

### **3.3.2 内置设计系统技能**

`match_design_system_skill` 命中时直接使用内置 Markdown（节省一轮 LLM），适用于常见行业风格（科技、医疗、电商等）。

### **3.3.3 Hero / Section 特效技能**

`heroSkillSelection` 按 `designIntent.keywords` 选择 `prompts/skills/section/hero/*.md`，向 Page Agent 注入 WebGL、粒子、滚动擦除等 **美化渲染** 约束（须遵守 SSR 安全与项目主题色，禁止照抄参考_palette）。

### **3.3.4 版式保真（截图模式）**

提供参考截图时，加载 `screenshotLayoutFidelity` 等 guardrail：约束间距、栅格、字号层级，使生成界面接近参考稿信息架构。

# **4. 系统逻辑**

## **4.1 引言**

### **4.1.1 编写目的**

阐述本系统在 **跨端前端界面智能生成** 与 **美化渲染** 方面的技术架构、数据模型与控制流，作为软著技术文档与维护依据。

### **4.1.2 解决的核心问题**

1. **界面风格漂移**：多区块由不同次 LLM 生成时颜色、字体、圆角不一致 → 用 Design Tokens + 规则束约束。  
2. **壳层重复**：每页各自写 Nav/Footer → Architect 先行统一 chrome。  
3. **不可运行交付**：仅输出代码片段 → 强制 `next build` 与 repair 闭环。  
4. **美化不可复用**：特效每次随机 → 技能库沉淀 Hero/section 模板。  
5. **预览与编辑割裂** → Studio 统一 SSE、预览、diff 与对话修改。

### **4.1.3 术语**

- **Blueprint**：页面与 Section 的结构化真源。  
- **Design Tokens**：`globals.css` 中的 CSS 变量主题。  
- **Chrome**：全站共享的导航/页脚等壳层组件。  
- **Section**：页面内相对独立的 UI 区块（Hero、Features 等）。  
- **Skill**：美化渲染用的 Markdown 技能模板。  
- **Guardrail**：注入 Agent 的强制规则（Tailwind 映射、TSX 输出格式等）。

---

## **4.2 总体架构**

```Plain Text
flowchart TD
    User[用户：自然语言 / 截图 / 风格 skill]
    User --> Intent[Intent Agent 可选]
    Intent --> Analyze[需求分析 + 设计意图]
    Analyze --> Plan[页面与 Section 规划]
    Plan --> DS[设计系统 / Token 美化]
    DS --> Arch[Architect 跨页壳层]
    Arch --> Pages[Page Implement 并行]
    Pages --> Assets[生图 + 依赖]
    Assets --> Build[类型检查 + next build + repair]
    Build --> Preview[跨浏览器预览渲染]
    Preview --> Modify[对话式界面修改]
```

---

## **4.3 核心模块**

### **4.3.1 界面蓝图子系统**

- `analyze_project_requirement`：产出 `ProjectBlueprint`（多页 `site.pages`、`experience.designIntent`）。  
- `plan_project`：扩展为 `PlannedProjectBlueprint`，为每 section 定义 layout/motion/visual traits。  
- `normalizeBlueprint`：容错 LLM 多种 JSON 形态，保障路由合法（`home`→`/`）。

### **4.3.2 美化渲染子系统**

- `match_design_system_skill` / `generate_project_design_system`  
- `apply_project_design_tokens` → `globals.css`  
- Hero skill 路由 + `framerMotionVariants` 等规则

### **4.3.3 界面生成子系统**

- `architect_agent`：`layout.tsx` + `components/chrome/**`  
- `page_implement_agent`：各页 `page.tsx` 与 `*_Section.tsx`  
- 文件命名 `{scope}_{Name}Section.tsx` 防止跨页冲突

### **4.3.4 预览渲染子系统**

- Storage 静态导出 + `/site-previews` 同源代理  
- E2B：`next build` → `serve out`  
- 修改后 `PUT .../preview` 触发重建

### **4.3.5 界面修改子系统**

- Agent Loop + Stop Hook  
- `search_code` 中文/英文关键词定位 Section  
- `edit_file` 精确替换，避免行号漂移  
- `FileSnapshotTracker` diff

## **4.4 控制算法要点**

### **4.4.1 页面并行调度**

```Plain Text
Algorithm: Parallel Page Implement
Input: PlannedProjectBlueprint.pages[], concurrency limit C
Output: merged generatedFiles[]

1. Partition pages into batches of size C
2. For each page p in batch (parallel):
     run page_implement_agent(p) → files_p
3. await Promise.all
4. Union files_p into generatedFiles
```

### **4.4.2 构建修复（界面自愈）**

```Plain Text
Algorithm: Build-Repair Loop
Input: generatedFiles, maxRounds=5
1. Run scoped TypeScript check; optional code-fix pass
2. Run next build
3. If success: return PASSED
4. Else extract error paths from stderr
5. repair_build targeted files only
6. Repeat until maxRounds
```

### **4.4.3 修改 Stop Hook**

```Plain Text
Algorithm: Modify Stop Gate
1. If no search and no edit → reject stop, suggest keywords
2. If edit but no build → reject stop, require run_build
3. If build failed → inject logs, continue loop
4. Else allow stop and compute diffs
```

## **4.5 安全与异常**

- **路径安全**：站点文件读写限制在 `sites/{projectId}` 根下。  
- **依赖沙箱**：`validate_dependency_scope` 类规则禁止危险 import。  
- **SSR 安全**：Hero 技能要求 WebGL/DOM 访问加 `typeof window` 守卫。  
- **URL 抓取**：参考站 digest 经 SSRF 校验后由服务端浏览器截取。

# **5. 常见问题**

**5.1 何谓「跨端」？**  
V1.0.0 以 **Web（Next.js）** 为主交付，界面采用响应式 Tailwind 布局，适配桌面与移动浏览器；蓝图与 Design Tokens 层可扩展至其他前端运行时。

**5.2 美化渲染是否会覆盖业务逻辑？**  
Hero/美化技能仅约束 **视觉与动效**；业务数据与交互须符合 Blueprint，禁止 skill 引入未需求的功能模块。

**5.3 参考截图与内置风格冲突怎么办？**  
有截图时优先 layout fidelity；内置 design-system skill 在截图流水线中可跳过，避免风格打架。

**5.4 生成失败如何排查？**  
查看 `logs/generate_project/{runId}/` 下各步骤日志与 `run_build/build-output.log`；Studio 展示失败步骤与摘要。
