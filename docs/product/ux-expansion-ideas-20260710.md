### 1.1 产品锚点（写想法时的约束）

- **定位**：从一句自然语言到可运行、可验证、可迭代的真实 Next.js 站点。
- **已有差异化**：生产级工程输出、透明流水线、Modify-first、Design Mode 源码写回、Community Publish Preview + Remix。
- **体验原则**：用户成功优先于功能数量；闭环优先；透明与可控。

## 2. 生成前：把「赌一把」变成「有把握」

### 2.1 可编辑的「站点蓝图预览」

生成前不只给文字纲要，而是出一版**低保真线框 + 模块清单**（Hero / Features / Pricing…）。用户可拖拽排序、删模块、改标题，再点生成。

| | |
|--|--|
| **爽点** | 贵 token 的生成变成「确认后执行」，抬首次可用率 |
| **与路线图关系** | 强化迭代大纲 **C2 生成前确认**；可演进为可交互 IA，而不只是只读纲要 |
| **建议优先级** | **P0**（优先押注之一） |
| **已拆 PRD** | [`generate-blueprint-preview-v0.1-prd.md`](./generate-blueprint-preview-v0.1-prd.md)（Intent Agent 确认层） |

### 2.2 「一句话 + 三问」Brief

用户丢一句 prompt 后，产品只追问 **3 个高杠杆问题**（受众 / 必有模块 / 气质参考），其余用智能默认。

| | |
|--|--|
| **爽点** | 比长表单轻，比裸 prompt 稳——对标「会问问题的设计师」 |
| **与路线图关系** | 轻量落地 **C1 Guided brief**；避免一上来重表单劝退 |
| **建议优先级** | P1 |

### 2.3 参考图「拆 section」向导

多图上传时自动建议：这张像 Hero、这张像 Pricing；置信度低时提示「再补一张细节图」。

| | |
|--|--|
| **爽点** | 把 image-to-code 的专业流程产品化，用户不用懂「一 section 一图」 |
| **与路线图关系** | 对齐 [`studio-visual-experience-v0.1-prd.md`](./studio-visual-experience-v0.1-prd.md) **P0-B Image Reference Pipeline** |
| **建议优先级** | P0（若视觉参考仍是主路径） |

## 3. Studio 内：改站像改稿，而不是像调 Agent

### 3.1 Design Mode「意图条」

点选元素后，除了色 / 字号 / 间距，加一条自然语言输入（例：「再大一点、更克制、像 Apple」）。点选上下文 + 这句话 → Direct Apply 或 Modify 草稿。

| | |
|--|--|
| **爽点** | 视觉点选 × 自然语言，相对纯聊天竞品的差异化 |
| **与路线图关系** | 建立在 Design Mode Direct Apply / Modify handoff 之上 |
| **建议优先级** | **P0**（优先押注之一） |

### 3.2 「改完立刻对比」双屏 Diff

每次 Apply / Modify 后，Preview 支持 Before/After 滑杆或并排，并标出改动区块。

| | |
|--|--|
| **爽点** | 消除「Agent 改了啥、我敢不敢继续」的焦虑 |
| **与路线图关系** | 强化迭代大纲 **D1 差异与说明**（从文件 diff 延伸到视觉 diff） |
| **建议优先级** | **P0**（优先押注之一） |

### 3.3 时光机式修改历史

按「意图回合」（Modify history turn）回放：每一步缩略图 + 一句话意图 + 一键回滚到该版本。

| | |
|--|--|
| **爽点** | 把工程语义的 history turn 做成用户能摸到的产品 |
| **与路线图关系** | 对齐 **D2 检查点与回滚**；术语见根目录 `CONTEXT.md`（Modify history turn） |
| **建议优先级** | P1 |

### 3.4 「说人话的构建失败」

Build 挂了不甩原始日志：用一句话解释原因 + 一个「让 AI 修」按钮 + 可选「我自己看 diff」。

| | |
|--|--|
| **爽点** | 失败从惊吓变成可恢复流程 |
| **与路线图关系** | 对齐 **E4 流水线对用户可见**、自动修复体验 |
| **建议优先级** | P1 |

## 4. 交付闭环：从「能预览」到「敢发出去」

### 4.1 分享前「上线体检」

一键跑：死链、对比度、移动端溢出、缺 alt、CTA 是否可见——结果用清单呈现，可勾选「自动修」。

| | |
|--|--|
| **爽点** | 分享前有「质检员」站岗，信任感拉满 |
| **与路线图关系** | 对齐阶段 **E**（关键路径 / a11y / Brief 对齐）的产品化入口 |
| **建议优先级** | P1 |

### 4.2 「给客户看的」只读链接 + 批注

在 Publish Preview 之上：访客可圈选「这里文案太长」并留言，作者在 Studio 收成 Modify 草稿。

| | |
|--|--|
| **爽点** | Community / Preview 从展示变成协作闭环——对自由职业者极杀 |
| **与路线图关系** | 延伸 Workspace / Community 模型；迭代大纲 §7「协作」的具体形态 |
| **建议优先级** | **P1**（优先押注之一：打开真实工作流） |

### 4.3 一键部署进度剧场

对接 Vercel 等时，用清晰阶段动画（打包 → 上传 → 域名 → 完成），失败给可操作下一步。

| | |
|--|--|
| **爽点** | 交付是情绪高潮，值得被设计成仪式感，而不是后台任务 |
| **与路线图关系** | 对齐阶段 **A1 / A2**（一键部署 + 部署引导）的体验层 |
| **建议优先级** | P1（依赖部署能力本身） |

## 5. Community / Remix：从逛变成「偷师」

### 5.1 Remix 后的「差异导览」

Remix 进来不是冷启动空白项目，而是：「原作结构是这样 → 你可以从改 Hero 文案开始」并带 3 个建议第一步。

| | |
|--|--|
| **爽点** | 降低 Remix 后的茫然，提高留存 |
| **与路线图关系** | 建立在 Publish Preview + Allow Remix + Remix lineage 之上 |
| **建议优先级** | P1 |

### 5.2 「同款气质」发现

按设计 token / 版式指纹聚类，而不是只按标签：点一个站 → 「气质相近的 6 个」。

| | |
|--|--|
| **爽点** | Community 从列表升级为灵感引擎 |
| **与路线图关系** | Community 发现体验；依赖可计算的视觉 / token 特征 |
| **建议优先级** | P2 |

### 5.3 作者「可 Remix 模块」

不只整站 Remix，允许标记「这个 Pricing 区块可单独拿走」。

| | |
|--|--|
| **爽点** | 生态颗粒度变细，创作者更愿意公开，使用者更敢试 |
| **与路线图关系** | 细化 Allow Remix；未来 monetization 可挂在模块级许可上 |
| **建议优先级** | P2 |

## 6. 长期记忆：让 Agent「越用越像你的设计师」

### 6.1 跨项目的品牌记忆

记住用户反复用的色板、语气（克制 / 热情）、禁用项（不要紫渐变、不要圆角胶囊）。新项目自动带上，可检视、可编辑。

| | |
|--|--|
| **爽点** | 直接打在「Agent 失忆」痛点上——叙事上最锋利的差异之一 |
| **与路线图关系** | 对齐 **C4 资产库** 的个人版；可先做用户级偏好，再做团队空间 |
| **建议优先级** | P2（基础设施重，但差异化强） |

### 6.2 「我的设计宪法」一页纸

用户可读的偏好页：已学到的规则 + 证据（来自哪次修改）。可删、可置顶。

| | |
|--|--|
| **爽点** | 记忆可检视 = 信任；黑盒记忆反而吓人 |
| **与路线图关系** | 6.1 的用户面；无检视 UI 则不做静默记忆 |
| **建议优先级** | P2（与 6.1 捆绑） |
