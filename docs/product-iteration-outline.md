# Open-OX 产品迭代大纲

> 本文档描述 **Open-OX**（AI-native 网站生产引擎）的中长期产品迭代方向，服务于最终用户：**从一句自然语言到可运行、可验证、可持续迭代的真实站点**。  
> 与工程实现细节相关的拆解见 `docs/architecture.md`；历史/更细条目可参考 `.planning/codebase/迭代方向.md`。

## 2. 产品定位与愿景

### 2.1 一句话定位

**Open-OX 是面向「要交付真实站点」的用户的 AI 生产引擎**：不是片段或截图生成器，而是带 **需求分析 → 设计系统 → 并行实现 → 构建验证与修复 → 沙箱预览 → 对话式修改** 的完整链路。

### 2.2 用户要解决的本质问题

| 用户诉求 | 产品回应 |
|----------|----------|
| 第一次生成就「大体对」 | 强意图引导、可确认的纲要/蓝图，减少盲目生成 |
| 生成后能改、改不脏 | Modify Agent + 差异可见 + 历史/回滚 |
| 跑得起来、能给人看 | 构建门禁、自动修复、可靠预览 |
| 能带走、能上线 | 导出工程包、一键部署与域名引导 |
| 心里踏实 | 可追溯步骤、用量与质量信号透明 |

### 2.3 愿景（12–24 个月）

成为 **「从 brief 到可部署 Next.js 站点」** 的默认选择：**个人与小型团队**可独立完成上品控站点；**团队**可复用品牌与设计系统资产，在受控成本下批量产出落地页与轻量站群。

## 3. 现状基线（迭代起点）

以下能力已在产品/仓库中具备或已定义约束，后续迭代在其上延伸：

- **生成管线**：需求分析、设计意图、项目规划、设计系统、架构 Agent、页面实现、构建、修复（多轮）。
- **运行时**：E2B 沙箱预览；Supabase 项目注册与存储。
- **修改回路**：工具型编辑 Agent、diff / 修改历史相关能力。
- **交付**：ZIP 导出（`GET /api/projects/[id]/export`）；本地/存储同步策略。
- **MVP 约束**：当前 **web profile 以单首页路由 `/` 为主**（意图层与流水线硬约束），多页需通过后续阶段显式产品化。

---

## 4. 迭代原则

1. **用户成功优先于功能数量**：每一步迭代必须能回答「用户哪类任务更可能一次成功」。
2. **闭环优先**：预览 → 导出 → **持久部署** 构成「可对外交付」的最小商业闭环。
3. **透明与可控**：黑盒生成不可持续；步骤、差异、成本信号要对用户可读。
4. **渐进松绑约束**：在系统可观测、可修复的前提下，**从单页扩展到多页/增量路由**，避免单次生成不可控膨胀。
5. **工程债可接受边界**：新能力必须配套 **构建/验收门禁** 或 **回滚路径**，避免「能生成不能维护」。

## 5. 战略主题（横切）

下列主题贯穿多个季度，按优先级在各阶段落地子项。

| 主题 | 说明 |
|------|------|
| **交付与运维** | 导出、部署、域名、环境变量与密钥引导 |
| **结构与规模** | 单页 → 多页、增量加路由、导航与 IA 产品化 |
| **意图与协作** | Guided brief、模板与复用、团队空间与权限 |
| **质量与信任** | 超越 build：验收清单、抽检、可访问性与关键路径测试 |
| **成本与公平** | 用量可见、队列、失败策略、分层套餐（若商业化） |
| **生态与扩展** | Skill/主题包、私有资产、API/自动化入口 |

## 6. 分阶段路线图

阶段划分以 **用户价值闭环** 为主轴；具体排期可按团队体量压缩或并行。

### 阶段 A：交付闭环强化（约 0–3 个月）

**目标**：用户能稳定完成「生成 → 预览 → **持久可访问** / **可带走工程**」。

| 方向 | 说明 | 优先级建议 |
|------|------|------------|
| A1 一键部署 | 对接 Vercel / Cloudflare Pages / Netlify 等之一或组合；从静态导出产物触发部署 | P0 |
| A2 部署引导 | 环境变量、自定义域名、常见问题（构建命令、Node 版本）的引导式 UI | P0 |
| A3 导出体验 | 强化导出与 Storage/本地一致性说明；大项目分卷或后台异步导出（按需） | P1 |
| A4 预览生命周期 | 沙箱过期预期、重建预览、失败重试的用户文案与状态机收敛 | P1 |

**阶段成功标准（示例）**：新用户能在无明显阻塞下获得 **24h+ 可分享 URL** 或 **可本地 `pnpm dev` 的 ZIP**；支持工单中「不知道怎么上线」类问题显著下降。

---

### 阶段 B：结构与规模（约 2–5 个月，可与 A 后半重叠）

**目标**：在可控复杂度下支持 **多页站点** 或 **清晰的增量扩展**，不牺牲构建稳定性。

| 方向 | 说明 | 优先级建议 |
|------|------|------------|
| B1 产品分层 | 明确产品形态：单页落地 / 小型多页 / 增量加页；引导与管线入口分离 | P0 |
| B2 增量路由 | 首页生成后，通过对话或表单「新增关于/定价/文档」等，走 architect + page 子流程 | P0 |
| B3 信息架构 | 全站导航、页脚、面包屑等与 blueprint 绑定，避免每页孤立生成 | P1 |
| B4 内容占位策略 | 真实文案 vs 占位符策略、多语言占位（若需要） | P1 |

**阶段成功标准（示例）**：典型「公司站」三页（首页+关于+联系）可在 **可接受的_token/时间** 内完成，且全站导航一致；单页模式仍为一键路径。

---

### 阶段 C：意图质量与复用（约 3–6 个月）

**目标**：降低重复生成；提升 **首次可用率** 与 **团队效率**。

| 方向 | 说明 | 优先级建议 |
|------|------|------------|
| C1 Guided brief | 分步表单/可选模板收集目标用户、必显模块、参考风格；与意图 Agent 对齐 | P0 |
| C2 生成前确认 | 用户可读的纲要（信息架构 + 关键模块），确认后再进入高成本生成 | P0 |
| C3 项目模板 | 从成功项目保存「蓝图 + 设计约束 + Skill 组合」；新项目一键套用 | P1 |
| C4 资产库 | 团队级设计 token、组件约定、文案口径（若有多用户/空间） | P2 |

**阶段成功标准（示例）**：同领域第二次建站 **人工编辑轮次** 明显下降；「模板起盘」路径有可量化采用率。

---

### 阶段 D：修改体验与信任（约 2–6 个月，持续）

**目标**：用户敢改、改坏了能救回。

| 方向 | 说明 | 优先级建议 |
|------|------|------------|
| D1 差异与说明 | 文件级 diff + 面向用户的「改了什么」摘要（非纯技术 diff） | P0 |
| D2 检查点与回滚 | 大改前快照、一键恢复某版本；与存储模型一致 | P0 |
| D3 分支或草稿（可选） | 实验性修改与合并（团队场景） | P2 |

**阶段成功标准（示例）**：误操作恢复路径清晰；支持对话中「撤销上一次」类高频诉求。

---

### 阶段 E：质量与可观测性（约 4–9 个月）

**目标**：从 **build 通过** 升级到 **对 brief 可用**。

| 方向 | 说明 | 优先级建议 |
|------|------|------------|
| E1 关键路径测试 | Playwright 等对核心 CTA、表单、链接的冒烟 | P1 |
| E2 Brief 对齐抽检 | 自动列出与 brief 冲突项（幻觉链接、缺失模块） | P1 |
| E3 a11y / 响应式 | 规则化检查与修复建议或自动修一轮 | P2 |
| E4 流水线对用户可见 | 步骤耗时、失败原因、修复轮数；与 Langfuse 等内部观测对齐展示策略 | P1 |

**阶段成功标准（示例）**：典型回归问题在 **预览前** 可被拦截或明示；用户感知的「死板错误」减少。

---

### 阶段 F：生态、成本与商业化（约 6–12 个月，按需）

**目标**：可持续运营；服务分层用户。

| 方向 | 说明 | 优先级建议 |
|------|------|------------|
| F1 Skill / 主题市场 | 可选视觉与动效包；私有 Skill 上传 | P1 |
| F2 API / 自动化 | CI/批处理触发生成或修改（B 端） | P2 |
| 用量与套餐 | Token/次数透明、队列优先级、失败不扣费策略等 | 视商业模式 |
| 企业版 | 私有部署、SSO、审计日志 | 按需 |

---

## 7. 项目与资产管理（跨阶段）

下列能力提升 **DAU 内的深度**与 **LTV**，可拆入各阶段小版本。

- **Studio 项目列表**：搜索、筛选、状态、排序、标签。
- **批量操作**：批量删除、批量导出（与合规/配额结合）。
- **协作（若多用户）**：共享项目、只读/编辑角色、评论或批注（远期）。

## 8.  Dependencies 与风险

| 风险 | 缓解思路 |
|------|----------|
| 多页导致上下文与成本暴涨 | 增量加页、分页线预算、纲要阶段锁定 IA |
| 部署平台差异大 | 先做 1 家深度集成 + 通用「静态导出 + 文档」退路 |
| 自动修复掩盖逻辑错误 | 验收测试 + 用户可见「已知问题」列表 |
| 沙箱与第三方配额 | 队列、降级为仅导出、预期管理 |

---

## 9. 成功指标体系（建议）

按阶段选用子集，避免一次性全上。

| 类别 | 指标示例 |
|------|----------|
| **激活** | 注册/首 prompt 到首次 `ready` 转化率 |
| **首次成功** | 首次生成无需人工再大改结构的比例（抽样评审） |
| **交付** | 导出/部署使用率、预览后 7 日回访 |
| **效率** | 每项目平均修改轮次、平均修复轮数 |
| **信任** | 回滚/重装率、客诉中「改坏了/打不开」占比 |
| **成本** | 每成功站点综合 token/计算成本（内部） |
