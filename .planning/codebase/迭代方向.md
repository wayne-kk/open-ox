## 一、核心体验层：生成质量与速度

### 1.1 增量预览（Hot Preview）

**当前问题**

任何修改都要走完整的 `next build → serve` 流程（30-60 秒）。改个颜色、改段文字也要等这么久。`rebuildDevServer` 复用了 sandbox 但仍需完整 build。

**迭代方向**

| 修改类型 | 检测方式 | 预览策略 | 预估耗时 |
|---------|---------|---------|---------|
| 纯文本修改 | diff 中只有字符串字面量变化 | 直接替换文件 + 浏览器刷新 | < 2s |
| CSS/className 修改 | diff 中只涉及 `className`、CSS 变量 | 替换文件 + 浏览器刷新 | < 2s |
| 组件结构修改 | diff 涉及 JSX 结构变化 | 替换文件 + `next build` | 30-60s |
| 新增/删除文件 | 文件列表变化 | 完整 rebuild | 30-60s |

- 在 `runModifyProject` 完成后，分析 `computeAllDiffs()` 的结果，判断修改范围
- 对于轻量修改，调用新的 `hotReloadSandbox(projectId, changedFiles)` 方法
- 长期方向：sandbox 内跑 `next dev` 而非静态导出，原生支持 HMR

**依赖项**

- 需要 E2B sandbox 支持文件级别的增量上传（当前 `uploadProjectToSandbox` 是全量上传）
- 需要在 sandbox 内注入一个轻量的文件监听 + 浏览器刷新机制

### 1.2 Section 生成质量评估与自动重试

**当前问题**

section 生成后只做了基础的静态检查（有 `export`、有 JSX return），但不检查视觉质量。生成出来的 section 可能结构正确但视觉效果差（如间距混乱、颜色不协调、响应式断裂）。

**迭代方向**

- 引入 **LLM-as-Judge** 评估环节：
  ```
  generate_section → static_check → quality_judge → (pass | retry with feedback)
  ```
- Judge 的评估维度：
  - 是否符合 `design-system.md` 的颜色/字体/间距规范
  - 是否有合理的响应式断点（`md:`, `lg:` 等 Tailwind 类）
  - 是否有足够的语义化 HTML（`<section>`, `<article>`, `<nav>` 等）
  - 是否有基本的无障碍属性（`aria-label`, `alt` 等）
- 评分低于阈值（如 < 7/10）的 section 自动重试，注入具体的改进建议
- 使用快模型（flash）做 judge，控制额外成本在 ~$0.001/section
- 优先从 hero section 开始（用户第一眼看到的），逐步扩展到所有类型

**度量指标**

- 引入前后的用户首次修改率（如果生成质量好，用户不需要立即修改）
- Judge 评分分布（目标：80% 的 section 首次生成即 ≥ 7/10）

## 二、Agent 能力层：修改 Agent 增强

### 2.1 多文件协同修改的上下文管理

**当前问题**

修改 Agent 最大 40 次迭代，复杂修改（如"把整个网站从浅色主题改成深色主题"）可能涉及 10+ 文件，容易接近模型的上下文窗口限制。每次工具调用的输入输出都会累积在 messages 中。

**迭代方向**

- 引入**修改计划阶段**（Plan → Execute 模式）：
  1. Agent 先用 `search_code` + `list_dir` 分析所有需要修改的文件
  2. 生成结构化修改计划（哪些文件、改什么、改的顺序）
  3. 按计划逐文件执行，每完成一个文件做 checkpoint
- **上下文压缩策略**：
  - 对已完成修改的文件，将完整内容替换为摘要（`"✅ home_HeroSection.tsx: +15/-8, 修改了标题颜色和按钮样式"`）
  - 工具调用结果超过 2000 字符时自动截断，保留头尾各 500 字符
  - `read_file` 结果在被 `edit_file` 使用后，从历史消息中移除完整内容
- 考虑引入 **sliding window** 机制：只保留最近 N 次工具调用的完整输出，更早的压缩为摘要

**技术方案**

```typescript
// 上下文压缩示例
function compressHistory(messages: ChatMessage[], maxTokens: number): ChatMessage[] {
  // 1. 保留 system prompt（不压缩）
  // 2. 保留最近 5 轮工具调用的完整输出
  // 3. 更早的工具调用：只保留 tool_name + 参数摘要 + 结果摘要
  // 4. 如果仍超限，移除最早的工具调用
}
```

### 2.2 视觉反馈闭环（Screenshot-in-the-Loop）

**当前问题**

修改 Agent 只能通过 `run_build` 验证代码正确性，无法验证视觉效果。用户说"把按钮变大一点"，Agent 改了代码但不知道改得够不够。当前架构已支持 `image_url` 格式的 vision 输入。

**迭代方向**

```
用户指令 → Agent 修改代码 → run_build ✅ → 自动截图 → Agent 自评
                                                          ↓
                                              满足需求 → 完成
                                              不满足  → 继续修改（最多 2 轮视觉迭代）
```

- 在 E2B sandbox 内安装 Puppeteer/Playwright，修改完成后自动截图
- 截图策略：
  - 全页截图（desktop viewport 1440×900）
  - 如果用户指令涉及特定区域，裁剪对应区域
  - 移动端截图（375×812）用于响应式验证
- 将截图以 `image_url` 格式注入 Agent 的下一轮消息
- Agent 对比截图与用户需求，决定是否需要继续修改
- 设置视觉迭代上限（建议 2 轮），避免无限循环

**成本评估**

- 每次截图 + vision 评估约消耗 ~1000 tokens（高分辨率图片）
- 每次修改增加 ~$0.01 成本，换取显著的修改准确率提升

### 2.3 Undo/Rollback 能力

**当前问题**

`FileSnapshotTracker` 已经在修改前做了快照（`this.snapshots.set(relPath, originalContent)`），但没有暴露 rollback API。用户说"改回去"只能靠 Agent 重新理解原始代码并手动还原。

**迭代方向**

- **Agent 侧**：新增 `rollback_file` 工具
  ```typescript
  // 新工具定义
  {
    name: "rollback_file",
    description: "将文件恢复到本次修改前的状态",
    parameters: {
      path: { type: "string", description: "要回滚的文件相对路径" }
    }
  }
  ```
- **前端侧**：在每条修改记录旁增加"撤销"按钮
  - 点击后调用 `POST /api/projects/[id]/rollback` 接口
  - 接口读取 `modificationHistory` 中对应记录的 diffs，反向应用
  - 撤销后自动触发 preview rebuild
- **持久化**：将快照写入 Supabase Storage（`snapshots/{projectId}/{timestamp}/{filePath}`）
  - 支持跨 session 的版本回退
  - 保留最近 10 个版本的快照，超出自动清理

### 2.4 修改 Agent 的专业化分工

**当前问题**

一个通用 Agent 处理所有类型的修改请求，从改文字到加新 section 到重构布局。system prompt 很长（包含 6 种工具的完整说明 + stop hook 逻辑 + 搜索策略），但每次只用到一部分。

**迭代方向**

引入 **Router → Specialist** 两层架构：

```
用户指令 → Router Agent（轻量分类，~500 tokens）
              ↓
    ┌─────────┼─────────┬──────────┐
    ↓         ↓         ↓          ↓
  Text      Style    Structure   Component
  Agent     Agent     Agent       Agent
  (改文字)  (改样式)  (改布局)    (加/删组件)
```

| Agent 类型 | 工具集 | 是否需要 build | 预估迭代次数 |
|-----------|--------|--------------|------------|
| Text Agent | `search_code`, `read_file`, `edit_file` | ❌ 跳过 | 3-5 |
| Style Agent | `search_code`, `read_file`, `edit_file` | ❌ 跳过（配合 Hot Preview） | 5-10 |
| Structure Agent | 全部 6 种工具 | ✅ 必须 | 10-20 |
| Component Agent | 全部 6 种工具 + `write_file` | ✅ 必须 | 15-30 |

- Router Agent 使用快模型（flash），分类延迟 < 1s
- 简单修改（Text/Style）的端到端时间从 ~30s 压缩到 ~10s
- 复杂修改仍走完整流程，不降级

### 2.5 新增 Section 与新增页面能力（高优先级）

**当前问题**

当前交互核心是“modify 现有内容”，对“新增内容”支持不足。用户常见需求如：
- “在首页新增一个 FAQ section”
- “新增一个 pricing 页面并接入导航”
- “新增案例页并复用现有设计系统”

这类需求如果只能靠通用修改，会导致：
- Agent 搜索定位困难（无目标文件可改）
- 文件创建、路由挂接、导航更新容易漏项
- 构建通过但信息架构不完整（页面孤立、入口缺失）

**迭代方向（P1）**

1. 新增意图识别（Add Intent Router）：
   - 在修改入口先判断请求类型：`modify` / `add_section` / `add_page`
   - 命中新增意图时，走专门流程而非普通 modify loop

2. 新增 Section 工作流：
   - 生成新 section 文件（复用现有 section 生成 prompt + skill）
   - 更新目标页面 `page.tsx` 的 section 组合
   - 自动做结构校验（导出、命名、引用存在）
   - 运行 build + 预览刷新

3. 新增页面工作流：
   - 生成新页面路由目录（如 `app/pricing/page.tsx`）
   - 生成该页面的 sections 和 compose 逻辑
   - 自动更新导航（`layout_NavSection.tsx` 或对应导航组件）
   - 可选：更新 sitemap/metadata（若项目已启用）
   - 运行 build + 预览刷新

4. 新增能力的强约束（防止“建了但不可达”）：
   - 新页面必须可从导航或明确入口访问
   - 新 section 必须在页面树中被引用
   - 新文件路径与命名必须通过规范校验
   - 失败时输出缺失项清单（不是笼统失败）

**建议接口与实现点**

- 新 API：
  - `POST /api/projects/[id]/add-section`
  - `POST /api/projects/[id]/add-page`
- 修改流程增强：
  - `runModifyProject` 增加 `intent` 分支
  - 新增 `plan_addition` 阶段，先给出要创建/修改的文件计划，再执行

**Slash 命令路由策略（建议纳入）**

- 在输入框支持三类显式命令：
  - `/modify`
  - `/add_section`
  - `/add_page`
- 路由规则：
  1. **有命令时：强制执行命令语义**（不再做意图猜测）
     - `/modify ...` → 仅走修改流
     - `/add_section ...` → 仅走新增 Section 流
     - `/add_page ...` → 仅走新增页面流
  2. **无命令时：自动意图识别**
     - 基于用户输入做 `modify | add_section | add_page` 分类
  3. **命令与内容冲突时：以命令为准**
     - 示例：`/modify 新增一个 pricing 页面`，仍按 modify 执行，并在 UI 提示“已按 /modify 执行”
- 前端交互建议：
  - 输入 `/` 弹出命令菜单（复用现有 slash menu 能力）
  - 命令执行前显示本次模式 badge（如 `Mode: add_page`）
  - 流式返回中附带 `intent_source` 字段：`"explicit_command"` 或 `"auto_classification"`
- 后端契约建议：
  - 请求体增加：
    - `intent?: "modify" | "add_section" | "add_page"`
    - `intentSource?: "explicit_command" | "auto_classification"`
  - 当 `intentSource="explicit_command"` 时，服务端禁止覆盖 `intent`

**场景化命令扩展（建议分阶段上线）**

- A. 结构编辑类（P1）
  - `/add_section`：向指定页面插入新 section（支持位置参数，如 `before:pricing`）
  - `/add_page`：新增页面并自动接入导航
  - `/remove_section`：删除指定 section，并清理无用 import
  - `/remove_page`：删除页面并同步移除导航入口
  - `/move_section`：跨页面移动 section 或调整页面内顺序

- B. 常规修改类（P1）
  - `/modify`：对现有内容进行修改（默认命令）
  - `/rename_section`：仅改 section 标题/文案语义，不变更结构
  - `/refactor_layout`：对布局结构做重排（强制 build 验证）

- C. 预览与验证类（P1）
  - `/preview`：触发预览刷新（支持 `light`/`full` 两种模式）
  - `/build`：仅执行构建与错误汇总，不做代码修改
  - `/verify`：执行质量检查（a11y/响应式/语义标签/设计系统一致性）

- D. 版本与恢复类（P1）
  - `/snapshot`：创建当前版本快照
  - `/rollback`：回滚到指定版本或最近一次修改前
  - `/diff`：比较当前与某版本差异（输出文件列表与 +/− 统计）

- E. 项目管理类（P2）
  - `/project_rename`：重命名项目
  - `/project_info`：查看当前项目结构摘要（页面、section 数量、最近修改）
  - `/export`：导出 zip（后续可扩展部署命令）

**典型使用场景示例**

- 场景 1（精准新增）：
  - 输入：`/add_section 在首页 hero 后新增 FAQ section，包含 6 条问答`
  - 预期：创建 section 文件 -> 挂接到首页 -> build -> 预览刷新

- 场景 2（结构化扩展）：
  - 输入：`/add_page 新增 Pricing 页面，包含方案对比和 FAQ，并加入导航`
  - 预期：新增路由与页面文件 -> 更新导航 -> build -> 可访问

- 场景 3（快速诊断）：
  - 输入：`/build`
  - 预期：不改代码，仅返回当前构建状态与错误定位建议

- 场景 4（安全回退）：
  - 输入：`/rollback latest`
  - 预期：回滚最近一次修改并自动触发预览刷新

**命令执行策略补充**

- 默认策略：
  - 无显式命令时，仍走自动意图识别，保持自然语言易用性。
- 参数缺失处理：
  - 命令必需参数缺失时，不进入执行，先返回参数提示（避免误操作）。
- 破坏性命令保护：
  - `/remove_page`、`/rollback` 执行前需要二次确认或 `--force` 标记。
- 降级策略：
  - 命令执行失败时返回可重试建议，并提供推荐替代命令（如 `/build` 先诊断）。
- 审计记录：
  - 每次命令执行写入 `modificationHistory`/trajectory，包含 `command`, `args`, `intent_source`。

**验收标准**

- 输入“新增 section”类需求，系统能在 1 次请求内完成创建、挂接、构建验证。
- 输入“新增页面”类需求，系统能完成路由创建、导航接入、构建验证。
- 90% 常见新增请求无需人工补改即可可预览。
- 显式命令场景下，路由结果与命令 100% 一致。

## 三、Skill 与设计系统层

### 3.1 Skill 库扩展

**当前状态**

- 全局风格 skill：4 个（`public/skills/` — minimal, bold, glassmorphism, brutalist）
- 组件级 skill：14 个（`ai/flows/generate_project/prompts/skills/` — hero ×6, features, pricing, navigation 等）
- 动效 skill：`prompts/motions/` 目录已存在但内容待充实
- 能力增强：`prompts/capabilities/` 目录已存在

**迭代方向**

| 扩展维度 | 具体内容 | 优先级 |
|---------|---------|-------|
| 行业垂直 skill | SaaS、电商、餐饮、教育、医疗、律所、创意机构等行业的设计规范与内容模板 | P1 |
| 动效 skill | scroll-triggered animations, hover effects, page transitions, loading states | P1 |
| 交互模式 skill | 表单验证、搜索过滤、tab 切换、modal 弹窗、toast 通知等常见交互 | P2 |
| 布局 skill | bento grid, magazine layout, asymmetric layout, full-bleed sections | P1 |
| 用户自定义 skill | 上传 brand guideline（PDF/图片/Figma URL），系统自动转化为 skill markdown | P2 |

- 用户自定义 skill 的技术路径：
  1. 用户上传 PDF/图片 → vision 模型提取设计规范
  2. 结构化为 skill markdown 格式（Visual Direction / Component Style / Tone）
  3. 存入 `user_skills/{userId}/` 目录，在 `preselect_skills` 时作为候选
- Skill 质量评估：建立 benchmark，对每个 skill 生成 10 个样本，人工评分，淘汰低质量 skill

### 3.2 设计系统的可视化编辑

**当前问题**

`design-system.md` 是纯文本 markdown，用户无法直观地调整颜色、字体、间距。修改设计系统需要通过自然语言描述，间接且不精确。

**迭代方向**

- 在 Studio 页面增加 **"Design System" 面板**：
  - 颜色面板：展示 primary/secondary/accent/background 等色板，支持 color picker 直接修改
  - 字体面板：展示 heading/body/mono 字体族，支持下拉切换 + 大小调整
  - 间距面板：展示 section-gap/container-padding 等间距值，支持滑块调整
  - 圆角/阴影面板：可视化调整 border-radius 和 box-shadow
- 修改 token 后的联动：
  1. 自动更新 `globals.css` 中的 CSS 变量
  2. 触发 Hot Preview（如果已实现 1.2）或完整 rebuild
  3. 更新 `design-system.md`（保持文本与可视化的同步）
- 支持从 **Figma** 导入 design tokens：
  - 通过 Figma REST API 读取 Variables / Styles
  - 映射为 Open-OX 的 design token 格式
  - 或支持 JSON 格式的 Design Tokens Community Group 标准导入

### 3.3 组件库与 Section 模板

**当前问题**

每次生成都是从零开始让 LLM 写 TSX，质量不稳定。同样的 "hero section"，不同次生成的质量差异很大。

**迭代方向**

- **Golden Templates 库**：
  - 从历史生成中筛选高质量 section，经人工审核后加入模板库
  - 每个模板包含：TSX 代码 + 截图 + 适用场景描述 + 质量评分
  - LLM 生成时参考模板结构，在此基础上做定制化修改（而非从零开始）
- **用户收藏机制**：
  - 用户可以"收藏"生成效果好的 section，加入个人模板库
  - 下次生成时，个人模板库中的 section 作为优先参考
- **社区共享**（长期）：
  - 用户可以发布自己的 section 模板供他人使用
  - 模板市场：按类型、风格、行业分类浏览
  - 评分与下载量排序

## 四、用户体系与协作

### 4.1 用户认证与多租户 🔴 P0

**当前问题**

架构文档明确指出："单用户模型，没有用户认证，所有项目共享同一个服务器文件系统。" 这是上线前的硬性阻塞项。

**迭代方向**

- **认证层**：接入 Supabase Auth（已有 Supabase 依赖，集成成本最低）
  - 支持 OAuth：Google、GitHub
  - 支持 Magic Link（邮箱登录）
  - 前端使用 `@supabase/auth-helpers-nextjs`，SSR 友好
- **数据隔离**：
  - `projects` 表增加 `user_id` 字段（FK → `auth.users`）

## 五、基础设施与可靠性

### 5.1 构建隔离 🔴 P0

**当前问题**

`run_build` 在主应用服务器上执行 `next build`，高并发时资源竞争严重（CPU/内存），且一个项目的构建失败可能影响其他项目的构建环境（如 node_modules 污染）。

**迭代方向**

- **方案 A：E2B 统一构建**（推荐）
  - 将 `stepRunBuild` 和 `stepRepairBuild` 迁移到 E2B sandbox 内执行
  - 生成阶段的 build 和预览阶段的 build 共用同一个 sandbox
  - 主服务器只负责 API 路由和 LLM 调用，完全不执行 `next build`
  - 额外好处：支持不同项目使用不同的 Node.js 版本和依赖版本
- **方案 B：Docker 容器化构建**
  - 每个项目的 build 在独立的 Docker 容器中执行
  - 适合自托管场景，不依赖 E2B
  - 需要维护构建镜像和容器编排
- **过渡方案**：
  - 引入构建队列（`p-queue`），限制同时构建的项目数（建议 max=2）
  - 每个项目的 build 在独立的临时目录中执行，build 完成后清理

---

### 5.2 LLM 调用的可观测性与成本控制

**当前问题**

`logs/` 目录有 artifact 日志，但缺乏系统化的监控。无法回答"上周花了多少钱在 LLM 上"、"哪个步骤最容易失败"等问题。

**迭代方向**

- **结构化日志**：每次 LLM 调用记录：
  ```typescript
  interface LLMCallLog {
    timestamp: string;
    projectId: string;
    stepName: string;
    modelId: string;
    promptTokens: number;
    completionTokens: number;
    totalTokens: number;
    latencyMs: number;
    status: "success" | "error" | "timeout";
    errorMessage?: string;
  }
  ```
  - 写入 Supabase 表 `llm_call_logs`，支持 SQL 查询
- **Dashboard**（内部工具）：
  - 按步骤维度：每个步骤的平均延迟、成功率、token 消耗
  - 按模型维度：不同模型的质量对比（配合 1.3 的 Judge 评分）
  - 按时间维度：日/周/月的成本趋势
  - 异常告警：单次调用 token > 阈值、连续失败 > 3 次
- **成本控制**：
  - 单个项目的 token 消耗上限（超过后降级到更便宜的模型或拒绝）
  - 用户级别的月度配额（配合 7.2 付费模型）
- **A/B 测试框架**：
  - 同一个步骤随机分配不同模型/prompt 版本
  - 对比生成质量（Judge 评分）和成本
  - 自动选择性价比最优的配置

### 5.3 E2B Sandbox 生命周期管理

**当前问题**

sandbox 有过期时间，用户重新访问时可能需要重建（30-60s 冷启动）。当前的 reconnect 逻辑在 `startDevServer` 中处理了三种情况，但缺乏主动的生命周期管理。

**迭代方向**

- **Sandbox 池化**：
  - 预热 N 个空白 sandbox（已安装 Node.js + Next.js 依赖）
  - 用户请求预览时，从池中分配一个，上传项目文件即可
  - 冷启动时间从 ~30s 压缩到 ~10s（省去 sandbox 创建 + 依赖安装）
- **智能清理**：
  - 超过 2 小时未访问的 sandbox → 暂停（保留状态，不计费）
  - 超过 24 小时未访问的 sandbox → 销毁（文件已在 Supabase Storage 中）
  - 用户重新访问时，从 Storage 恢复文件到新 sandbox
- **持久化预览**：
  - 对于"已发布"的项目，将 `/out` 静态文件部署到 CDN
  - 不再依赖 sandbox，URL 永久有效
  - sandbox 仅用于开发/修改阶段

### 5.4 错误恢复与重试策略 🔴 P0

**当前问题**

LLM 调用失败时的重试策略比较简单：section 生成失败 retry 一次，build repair 最多 2 轮。没有处理 rate limit、provider 故障等场景。

**迭代方向**

- **指数退避重试**：
  ```typescript
  // 重试策略
  const retryConfig = {
    maxRetries: 3,
    baseDelayMs: 1000,
    maxDelayMs: 30000,
    retryableErrors: [429, 500, 502, 503, 504, "ETIMEDOUT", "ECONNRESET"],
  };
  // delay = min(baseDelay * 2^attempt + jitter, maxDelay)
  ```
- **模型 Fallback 链**：
  ```
  gemini-3-flash-preview → gpt-5.2 → gemini-3.1-pro-preview
  ```
  - 主模型连续失败 2 次后，自动切换到 fallback 模型
  - 通过 SSE 通知前端："已切换到备用模型"
- **优雅降级**：
  - 某个 section 生成失败 → 跳过，继续生成其他 section → 最后告知用户哪些 section 缺失
  - `preselect_skills` 失败 → 所有 section 使用 fallback skill
  - `generate_project_design_system` 失败 → 使用内置的默认设计系统
- **错误分类与上报**：
  - 区分可重试错误（rate limit、网络超时）和不可重试错误（invalid API key、模型不存在）
  - 不可重试错误立即终止并给出明确的错误信息

## 九、关键技术债务

在推进新功能之前，以下技术债务需要关注：

| 债务 | 影响 | 建议处理时机 |
|-----|------|------------|
| 模型配置使用全局变量（`_runtimeModelId`、`_stepModelMap`） | 并发请求间会互相覆盖模型配置 | P0（与构建隔离一起解决） |
| Supabase 客户端使用 publishable key（无 RLS） | 任何人可以读写所有数据 | P0（与用户认证一起解决） |
| `sites/` 目录在主服务器本地 | 服务器重启/迁移会丢失文件 | P0（依赖 Storage 恢复，但需验证完整性） |
| `devServerManager.ts` 中的 sandbox 操作无超时保护 | E2B 服务异常时会 hang | P1 |
| `runModifyProject` 的 messages 数组无上限 | 40 次迭代可能产生 100K+ tokens 的历史 | P1（与 2.1 一起解决） |
| 无请求级别的 tracing（request ID） | 无法关联一次用户操作的所有日志 | P1（与 5.2 一起解决） |
