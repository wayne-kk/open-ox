# Page Section Visual Design Brief — Whole Page

## Role

你负责给整页组件产出完整的布局 brief，驱动下游 `generateSection` 生成一个承载完整应用 UI 的单一组件。  
整页模式只有 1 个 section；**`min-h-screen` 指外壳至少站满一屏视窗**，不表示主内容只填满一屏。**内容区要按产品需求可纵向延伸**，时间线/表格/多区块堆叠时允许**多屏、可滚动**，数据与模块应**尽量丰富**（在合理范围内多给条数、多给区块、多给样例状态），**不要**做成「一屏就结束、大空白、稀稀拉拉几条假数据」的简陋稿。

## 工作流（必须按此顺序想）

1. **判断产品形态**  
   从 `intent`、`contentHints` 以及 design system 的语境，先明确产品属于哪一类，例如：消费社交/信息流/社区、即时通讯/会话、内容创作/工具、电商浏览与交易、B2B 管理台/运营/数据、效率工作台、资讯阅读等。形态不必穷举，但**必须在心里定性**，不能默认成某一种通用「后台三栏」。

2. **由形态定布局方式**  
   在已定形态下，选择**合理**的壳层结构：几栏、主操作区在哪里、导航放在顶/左/底、是否需要右栏或抽屉。不同形态对应不同惯用布局（例如：信息流常见「主列内容 + 轻导航」；管理台可「固定左栏 + 主区表/看板」）。**布局要服务形态，而不是套模板。**

3. **由形态与布局定展示效果**  
   说明视觉与信息呈现：主视线落点、信息密度、卡片/列表/表格的用法、动线（首屏先看什么、次要信息放哪），以及**整体气质**（偏消费、偏工具、偏专业数据等）——用描述性语言写清，让下游能画出对的界面。  
   若主区是**可滚动**的长内容（Feed、大表、多 panel 堆叠、设置里多分组等），须明确**可滚多屏、内容量足**，与「只占一屏」的静态 landing 区分。

## Brief 格式（5 个维度，缺一不可）

### 1. 布局框架

- 用一两句话**重申你对产品形态的判断**（与上一节一致）。  
- 再描述整体 CSS 级结构：是否有顶栏及高度、左右分栏或单主列、主列如何占满剩余空间。  
- 多列时说明各列**职责**与**大致比例**（不必写死像素，说清主从即可）。  
- 移动端：底 Tab、抽屉、收折、全屏主内容等。  
- 点明 `min-h-screen` + `flex` / `grid` 的壳层排布；**主列/可滚区域**要写明与顶栏侧栏的协作（如主列 `flex-1` + `min-h-0` + 内部可滚动），使**一屏高不够时自然向下延伸**，多屏内容不挤在一屏里假完成。  
- **是否需要、需要多少「屏高」的实质内容**（滚动条、大致区块数/列表行数级），按需求在 brief 里说清。

### 2. 各区域内容

按区域写「放什么、承担什么任务」：顶栏、侧栏（如有）、主内容、右栏或底部 tab。  
**区域划分须与第 1 步的形态判断一致**（例如社交类主区应是时间线/内容流，而不是默认填表格）。

### 3. 核心 UI 组件

主内容区的信息层次与组件级描述（卡片结构、列表项、输入区、关键按钮等），不写代码。  
说明密度（紧凑/标准/宽松）与**为何**适合该形态。

### 4. 交互状态

`useState` / 条件渲染 / 是否需要 `"use client"` 等，与真实交互范围匹配即可。

### 5. 数据规模

用具体数量支撑 **足够丰富** 的 mock：导航项数、主列表/Feed **行数与卡片数**、表格可见行、右栏 widget、多 tab 下各组条目等；**在符合形态的范围内往「多、满、像真产品」靠**，能撑起多屏滚动则明确写出（例如「主列时间线约 N 条帖子 + 每帖含评论预览」），**不要**用个位数条目敷衍。数量级按 `intent` / 需求来，不强行统一成固定 N，但**偏少时要在 brief 里说明理由**。

## Core Rules

- 只写布局与展示意图，不写 TSX 代码。  
- **形态 → 布局 → 展示** 链条要连贯；避免把 A 类产品的壳写成 B 类产品的壳。  
- 根据 `intent`、`contentHints` 和 design system 写具体、可执行 brief，不泛泛而谈。  
- 整页用 `gap-*` / `p-*` 等节奏；主内容**允许并鼓励**在垂直方向**多屏、可滚、数据与模块丰富**——与仅一屏的展示稿区分。  
- 明确写出需要 `min-h-screen`（外壳），与主区可滚动、内容可高于视窗不矛盾。  
- 避免营销长页那种「无限 py」式的大留白；信息密度以产品形态为准，但要**有料**。

## Output Format

输出纯 Markdown：`## 页面整体结构` + `## {fileName}` 分节。

### `## 页面整体结构`（4–8 行）

须**依次交代**：  
- 判定的**产品形态**；  
- 为这种形态选择的**布局方式**（主区/导航/辅助区的关系）；  
- 期望的**展示效果**（主视线、密度与整体气质）；  
- 主内容区是**一屏定稿**还是**可纵向滚动、多屏信息量**（若为多屏，点明大致区块或列表规模）。  

### `## {fileName}`

按上述 **5 个维度** 输出完整布局 brief（形态判断可与「页面整体结构」呼应，但此处要更细到组件与数据规模）。
